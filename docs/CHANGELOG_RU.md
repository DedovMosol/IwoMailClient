### Аудит серверных черновиков и виджета — часть 4 (12.07.2026)
- **Сирота-черновик после отправки** (D-1): при отправке серверного черновика он удалялся только из локальной БД — на сервере оставался сиротой (виден в Outlook/OWA) и «воскресал» при следующей синхронизации папки Drafts. Теперь `SendProgressBar` вызывает `mailRepo.deleteDraft` (серверное EWS-удаление + локальная очистка + `registerDeletedEmail` от воскрешения) в фоне, не задерживая звук/навигацию успеха. DRY — переиспользует тот же метод, что путь пере-сохранения
- **Удаление всех вложений черновика** (D-3): если черновик имел вложения, а пользователь удалил их все, шёл путь `updateDraft` (EWS `SetItemField`), который по [MS-докам](https://learn.microsoft.com/en-us/exchange/client-developer/web-service-reference/updateitem-operation) не умеет менять вложения — удалённые вложения оставались на серверном черновике. Условие delete+create расширено на `email.hasAttachments` (исходный черновик имел вложения), с корректным `hasAttachments` у нового черновика. На Exchange 2007 пустой набор идёт в `createDraftEws` → валидный черновик без вложений
- **DRY цветов темы виджета** (W-2): `themeToColor()` хардкодил hex-цвета, дублируя `AppColorTheme` (Theme.kt) — риск дрейфа при смене палитры. Сведено к единому источнику: `AppColorTheme.fromCode(code).gradientStart.toArgb()` (точный round-trip, PURPLE по умолчанию = прежней ветке else)
- **Тесты форматтера виджета** (W-3): чистая логика метки «время синка» вынесена в `formatSyncLabel` (без `Context`) + 6 юнит-тестов на все ветки и края (нет синка, <60с, ровно 60с, часы уехали назад, не-сегодня → дата, сегодня → «в/at HH:MM»); `formatSyncAgo` делегирует
- **Тесты**: сьюта 438 (было 432)

### Аудит «лёгкие победы» — часть 2 (10.07.2026)
- **Обрезка HTML-писем на первом `--`** (A-12): извлечение части из MIME остановилось на первом `--` в любом месте тела — на Outlook conditional comments (`<!--[if mso]>`), `<style><!--` и `-----Original Message-----`, теряя весь остаток письма. Основной путь переписан на честный RFC 2046-сплит по boundary (обрезка по содержимому в принципе невозможна), regex-fallback оставлен для одночастного MIME; рекурсивный спуск во вложенные multipart. Покрыто `MimeHtmlProcessorExtractHtmlTest` (9 тестов, включая Outlook-комментарии и подпись `-- `)
- **NTLM-аутентификация EWS через пул соединений** (A-13): 401-челлендж дочитывается полностью (непрочитанное тело заставляло OkHttp выбросить соединение, и Type3 уходил по новому TCP → плавающие 401, т.к. NTLM аутентифицирует соединение); handshake выделен в отдельный клиент с `ConnectionPool(1)`, чтобы параллельные EAS/EWS-запросы не «крали» аутентифицируемое соединение между Type1 и Type3
- **«Сохранить как…» при повороте** (A-14): id вложения для системного файл-пикера переведён на `rememberSaveable` — поворот/смерть процесса поверх пикера больше не отменяют сохранение молча (объект резолвится из списка вложений по id)
- **Диалог автообновления при повороте** (A-15): `UpdateInfo` сохраняется через `listSaver` — диалог доступного обновления переживает поворот экрана, а не исчезает до следующей проверки по интервалу
- **Ping Status=5: серверный HeartbeatInterval** (A-16): по MS-ASCMD при выходе интервала за пределы сервер возвращает граничное допустимое значение — принимаем его (в наших рамках 120–1800 с) сразу, вместо медленной сходимости делением пополам
- **Base64 вложений с CRLF** (A-17): строки base64 в MIME-вложениях разделяются CRLF (RFC 2045), а не голым LF (`Base64.DEFAULT`) — единообразно с телом письма
- **DRY диалогов очистки корзины** (A-18): три идентичных каркаса диалога (задачи/заметки/календарь) заменены общим `EmptyTrashConfirmDialog` — специфика действия остаётся в колбэке экрана
- **DRY MIME-детекта Content-Type**: повтор `contains("Content-Type: x") || contains("Content-Type:x")` (6 мест) вынесен в `String.hasContentType(prefix)`
- **Тесты**: сьюта 430 (было 421) — новый `MimeHtmlProcessorExtractHtmlTest`

### Отложено (следующие итерации, вне рамок «лёгких побед»)
- **HTTP 451 (ActiveSync redirect, `X-MS-Location`)**: корректная поддержка требует мутации серверного URL, инвалидации lazy `ewsUrl` и персиста в БД — полукорректная версия оставила бы `ewsUrl` рассинхронизированным. Частично компенсируется ручным dual-URL (`EasClient.probeAlternateUrl`)
- **markAsReadViaEws по теме** (EWS FindItem по `item:Subject` в 4 папках): два письма с одинаковой темой могут получить общий статус прочитано — сузить до `InternetMessageId`/даты либо включать fallback только при подтверждённом отказе EAS-пути
- **Async-конвейер тела письма**: разбор MIME/HTML до 20 МБ вынести в `produceState`/`Dispatchers.Default` (сейчас под `remember`, но сравнение многомегабайтной строки на кадр)
- **Тесты протокольного ядра**: `WbxmlParser.parse/generate`, `NtlmAuthenticator` (NTLMv2), recurrence-логика календаря — ноль покрытия; требуется тест-seam
- **MVVM для `MainScreen`/`ComposeScreen`**: последние крупные немигрированные экраны (см. `COMPOSESCREEN_AUDIT.md`)
- **EWS auth-fallback**: уже DRY за deps-лямбдами; 6-строчная обёртка с разными контрактами (`String?` vs `EasResult<String>`) осознанно не объединялась
- **Дружелюбный auth-тост при отправке**: сейчас ошибки отправки показываются raw-сообщением сервера; можно детектировать auth-ошибку и показывать локализованный текст (низкий приоритет, не баг)
- ✅ **CI** — сделано в части 3 (GitHub Actions + Kover 0.7.6)

### Архитектура — внедрение MVVM (инкрементально)
- Введён слой `ViewModel` + неизменяемый `UiState` + `StateFlow` для устранения багов с потерей состояния при повороте экрана
- `SyncCleanupScreen` → `SyncCleanupViewModel`: настройки синхронизации и очистки переживают поворот без ручных флагов
- `SearchScreen` → `SearchViewModel`: результаты поиска, выделение и batch-операции живут в ViewModel; убрана хрупкая связка `rememberSaveable` + повторный запрос в БД (снят риск `TransactionTooLargeException`)
- `NotesScreen` → `NotesViewModel`: заметки/корзина/выделение/поиск и авто-синхронизация живут в ViewModel; убран флаг `dataLoaded`, который при повороте/смене аккаунта мог приводить к лишней синхронизации; операции с прогресс-баром (окончательное удаление/восстановление/очистка корзины) инкапсулированы за suspend-методами
- `TasksScreen` → `TasksViewModel`: задачи/корзина/фильтр/выделение/поиск и авто-синхронизация живут в ViewModel; убран флаг `dataLoaded`; пакетные операции восстановления и окончательного удаления (с прогресс-баром) перенесены из Composable в suspend-обёртки ViewModel
- `UserFoldersScreen` → `UserFoldersViewModel`: список пользовательских папок (фильтр/сортировка), выделение, создание/переименование/удаление и пакетное удаление с прогрессом живут в ViewModel; долгие операции перенесены из `rememberSyncScope` (отменялся при повороте) в `viewModelScope` — теперь переживают поворот
- `EmailListScreen` → `EmailListViewModel`: список писем (папка/избранное/«Сегодня»), фильтры, выделение, синхронизация и пакетные операции (в корзину/спам, перемещение, восстановление, прочитано/непрочитано, флаги) живут в ViewModel; операции перенесены из `rememberCoroutineScope` (отменялся при повороте) в `viewModelScope` — теперь не прерываются при повороте; авто-синк «Черновиков» выполняется ровно один раз (без повторного синка и гонок при повороте); текущая папка и список папок для переноса берутся из единого реактивного потока
- `EmailDetailScreen` → `EmailDetailViewModel` (ядро экрана): письмо/вложения/папки, загрузка тела и inline-картинок, операции (обновление, удаление в корзину, перенос, восстановление, прочитано/непрочитано, флаг, MDN) живут в ViewModel; операции перенесены из `rememberCoroutineScope` (отменялся при повороте — обрыв удаления/переноса) в `viewModelScope`; inline-картинки кэшируются в состоянии — устранена их повторная сетевая загрузка при каждом повороте; UI-only (WebView, вложения, iCal/задачи, приглашения на встречи) остаётся в Composable
- Одноразовые UI-события (тосты, звук, пасхалка) вынесены в канал событий (`Channel` + `receiveAsFlow`) — ViewModel не зависит от локализации/ресурсов
- Зависимости передаются через конструктор (DIP): системные эффекты push/синхронизации скрыты за интерфейсом `SyncEffects`

### Стабильность и безопасность (аудит кода)
- Устранены краши при изменении настроек синхронизации/очистки и в операциях с заметками/задачами/папками/поиском: обязательный перехват исключений с корректным сбросом флагов загрузки/выделения (M-1)
- Исправлен краш Push-сервиса на Android 12+ (`ForegroundServiceStartNotAllowedException`): безопасный запуск переднего сервиса + graceful-остановка (N-10)
- Восстановлен watchdog Push на Android 8+: ресивер экрана/разблокировки/зарядки регистрируется динамически (из манифеста он не срабатывал → деградация Push) (N-12)
- Предотвращён OOM при отправке крупных вложений: суммарный размер проверяется ДО чтения в память (лимит 10 МБ + уведомление) (N-2)
- Редактор письма: Content-Security-Policy + единый HTML-санитайзер вырезают inline-обработчики `on*`/`javascript:` (в т.ч. при вставке из буфера) (L-3)
- Устранена гонка кэша EAS-клиента при переключении на резервный сервер (N-7)
- Дедупликация исходящих: стабильный `ClientId`/`Message-ID` предотвращает повторную отправку при ретрае из очереди (N-11)
- Защита от инъекции MIME-заголовков: адреса и Message-ID во всех путях сборки письма (отправка, приглашения на встречи, отчёты о прочтении MDN) очищаются от CR/LF единым санитайзером; критично для MDN, где значения берутся из входящего письма (N-1)
- Корректная отправка длинных тем: Subject сворачивается по RFC 2047 (несколько encoded-word'ов) — строка заголовка не превышает лимиты и не отклоняется строгими шлюзами; единый кодировщик для всех путей сборки письма (N-3)
- Безопасная дедупликация писем: ключ очистки дублей переведён на канонический Message-ID — исключено удаление разных писем с совпавшими темой/отправителем/временем (L-4)
- Exchange только по HTTPS: убрана нерабочая HTTP-опция при настройке Exchange-аккаунта (cleartext блокировался системной политикой безопасности), для Exchange принудительно включается SSL; IMAP/POP3 не затронуты (N-13)
- Единый разбор inline-картинок писем: устранено дублирование логики извлечения из MIME (один источник), добавлена защита от «зловредного» глубоко-вложенного MIME (N-5)
- Энергосбережение: все экраны переведены на lifecycle-aware сбор данных (`collectAsStateWithLifecycle`) — сбор потоков приостанавливается, когда экран не виден, снижая расход CPU/батареи в фоне (N-8)
- Устойчивость к сбоям: операции в экранах (удаление аккаунтов/контактов/событий, настройки и пр.) больше не могут уронить приложение при ошибке БД/ввода-вывода — сбой изолируется и логируется (UI-1)
- Настройка аккаунта: выбранный сертификат (серверный/клиентский) больше не удаляется при повороте экрана во время настройки — Exchange с самоподписанным сертификатом настраивается корректно (SET-1)
- Верификация аккаунта: введённый пароль больше не сбрасывается при повороте экрана во время проверки — устранено ложное «сеанс проверки истёк, введите пароль заново» (SET-2)
- Вложения: убран сбойный fallback, из-за которого при неудаче прямого скачивания сохранялось всё письмо целиком вместо самого вложения (повреждённый файл) (PB-1)
- Стабильность: открытие «тяжёлых» писем с крупными вложениями/картинками больше не роняет приложение при нехватке памяти — письмо показывается без встроенных картинок (PB-2)
- Устойчивость к сбоям (продолжение): фоновые операции приложения, push/синхронизации, будильников и ресиверов больше не могут уронить процесс при непойманной ошибке — единый крах-безопасный scope изолирует и логирует сбой
- Устойчивость к сбоям (реактивные экраны): наблюдение за письмами/папками/заметками/задачами больше не может уронить приложение при ошибке базы данных — сбой изолируется и логируется (CR-2)
- Календарь: корректный учёт часового пояса при разборе времени событий/задач, если сервер вернул смещение вместо UTC — устранён возможный сдвиг времени (N-4)
